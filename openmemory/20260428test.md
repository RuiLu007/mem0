## 一、测试概况

### 测试目标

验证 OpenMemory 在**无 API Key（Mock 模式）**和**有 Key（生产模式）**两种条件下，完整支持酒店客服微信聊天记录的写入、检索与分类。

### 业务场景

模拟一位酒店客服与客户通过微信进行的完整服务闭环，覆盖 5 大典型场景：

| 场景 | 条目数 | 典型内容 |
|------|--------|---------|
| **预订咨询 (booking)** | 6 条 | 询问空房、房型偏好（高楼层/无烟/大床）、报价、确认预订 |
| **入住办理 (checkin)** | 6 条 | 提前入住、加床服务、押金支付方式 |
| **设施询问 (facilities)** | 8 条 | 游泳池/健身房/早餐/停车场时间、收费、位置 |
| **投诉处理 (complaint)** | 6 条 | 空调噪音投诉、赔偿协商、代金券发放、工单记录 |
| **退房结算 (checkout)** | 6 条 | 账单核查、餐饮消费确认、增值税发票申请 |

合计对话条目：**32 条**（客户+客服双向）

## 二、阶段一成果：代码重构说明

### 2.1 QWEN_API_KEY 别名支持

**新增字段 `api_key_env_aliases`（`providers.py`）：**

```python
@dataclass
class ProviderDef:
    api_key_env: str             # 主环境变量名 (DASHSCOPE_API_KEY)
    api_key_env_aliases: List[str] = field(default_factory=list)  # 别名列表

# Qwen 配置
"qwen": ProviderDef(
    api_key_env="DASHSCOPE_API_KEY",
    api_key_env_aliases=["QWEN_API_KEY"],   # ← 支持双名称
    ...
)
```

**更新后的优先级链：**
```
resolve_api_key('qwen')
  ① 检查 DASHSCOPE_API_KEY（主变量）
  ② 检查 QWEN_API_KEY（别名）
  ③ 返回空字符串 → 触发 Mock 模式
```

### 2.2 新增 `is_real_api_key_available()` 函数

```python
def is_real_api_key_available(provider_name: str) -> bool:
    key = resolve_api_key(provider_name)
    return bool(key) and not key.startswith("sk-mock") and key != "no-key"
```

### 2.3 Mock 模式自动检测

在 `memory.py` 的 `get_default_memory_config()` 中，当 `MOCK_MODE` 未设置时，自动检测：

```python
if explicit_mock == '':
    if not is_real_api_key_available(llm_provider_alias):
        os.environ['MOCK_MODE'] = 'true'   # 无 Key → 自动 Mock
    else:
        os.environ['MOCK_MODE'] = 'false'  # 有 Key → 生产模式
```

### 2.4 Mock 模式降级链（完整流程）

```
用户调用 POST /api/v1/memories/
         │
         ├─ [1] MOCK_MODE=true ?
         │        ↓ YES
         │   use_infer = False  (跳过 LLM 事实提取)
         │        ↓
         ├─ [2] memory_client.add(text, infer=False)
         │   使用本地 HuggingFace Embedder：
         │   sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2
         │        ↓
         ├─ [3] 写入 FAISS 索引（384 维）
         │   faiss_store/openmemory.faiss  + openmemory.json
         │        ↓
         ├─ [4] 写入 SQLite：memories 表
         │        ↓
         └─ [5] 关键词规则分类（categorization.py: _mock_categorize）
                  无需调用任何外部 API
```

### 2.5 生产模式（有 QWEN_API_KEY）

```bash
# .env 切换
MOCK_MODE=false         # 或删掉此行，auto-detect 会检测到 key
LLM_PROVIDER=qwen
QWEN_API_KEY=sk-xxx     # 或 DASHSCOPE_API_KEY=sk-xxx
```

切换后：
- LLM 调用：`qwen-plus`（通过 DashScope OpenAI-compatible API）
- Embedding：`text-embedding-v3`（DashScope）
- 记忆提取：通过 Qwen 智能分析对话，提炼关键事实

## 四、模型思考过程与技术决策

### 决策 1：QWEN_API_KEY 设计方案

**问题**：用户可能用 `QWEN_API_KEY` 或 `DASHSCOPE_API_KEY` 两个名称。

**候选方案**：
| 方案 | 优缺点 |
|------|--------|
| 只支持一个名称 | 简单但用户体验差 |
| 在代码里写死两个变量 | 不可扩展 |
| **ProviderDef 增加 `api_key_env_aliases` 列表** ✅ | 可扩展，不破坏现有接口 |

**决策**：选方案3 —— 给 `ProviderDef` 添加 `api_key_env_aliases: List[str]` 字段，`resolve_api_key()` 先检查主变量，再遍历别名列表。未来新增 provider 时，可轻松为一个 key 添加多种别名。

### 决策 2：Mock 模式 Embedder 选择

**问题**：`Memory.add()` 必须使用 Embedder 生成向量才能写入 FAISS。OpenAI Embedder 需要真实 API Key。

**候选方案**：
| 方案 | 优缺点 |
|------|--------|
| 完全跳过向量化，只写 SQLite | 丢失向量搜索能力 |
| 使用 fastembed（轻量） | 未安装，需额外安装 |
| **HuggingFace sentence-transformers** ✅ | 已安装，支持中文，384维 |
| 使用 Ollama 本地模型 | 需要 Ollama 服务 |

**决策**：选 HuggingFace。`paraphrase-multilingual-MiniLM-L12-v2` 多语言支持，中文效果好，已缓存在服务器，启动快。

**注意点**：需要同步修改 FAISS 向量维度从 1536（OpenAI） 到 384（MiniLM），并清理旧索引。

### 决策 3：LLM 提取 vs `infer=False`

**问题**：mem0 的 `Memory.add()` 默认调用 LLM 提取关键事实（`infer=True`）。Mock 模式下 LLM 调用失败，返回空结果。

**候选方案**：
| 方案 | 优缺点 |
|------|--------|
| 全局禁用 infer | 生产模式也无法提取事实 |
| 使用本地 Ollama LLM | 需要 Ollama 服务 |
| **MOCK_MODE 时自动 `infer=False`** ✅ | 不影响生产模式，存原文，分类靠关键词规则 |

**决策**：在 `memories.py` 的 `create_memory` 中判断 `MOCK_MODE`，是则覆盖 `use_infer=False`。这样：
- Mock 模式：存储完整对话原文 → 关键词分类 → FAISS 向量化（HF本地）
- 生产模式：LLM 提取关键事实 → 智能分类 → FAISS 向量化（云端 Embedder）

### 决策 4：用户创建问题

**问题**：`POST /api/v1/memories/` 需要用户预先存在于 SQLite，但 OpenMemory 默认只在启动时从 `USER` 环境变量创建单一用户。

**解决**：
1. 种子脚本优先使用已存在的 `hotel_guest` 用户（由 `.env USER=hotel_guest` 在启动时自动创建）
2. 提供 `--db` 参数支持直接向 SQLite 插入测试用户（仅限本地开发）
3. 生产场景中，企业微信接入时应通过 MCP/API 先注册用户
